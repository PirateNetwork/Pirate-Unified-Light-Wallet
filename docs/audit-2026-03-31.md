## Method

This was a source audit. The review focused on:

- key and seed handling
- wallet secret storage
- transaction building and value accounting
- sync integrity and transport trust
- API/FFI exposure of high-risk wallet functions
- update and release authenticity
- privacy-sensitive logging and clipboard handling


## Summary

The codebase contains several material issues that should be fixed before relying on the current state as a hardened wallet stack.

The most important findings are:

1. seed and spending-key export protections are inconsistent and can be bypassed from public wallet surfaces
2. wallet secret storage can accumulate duplicate rows for the same logical wallet, creating nondeterministic secret selection
3. deleted wallets can remain accessible through stale database handles
4. the default lightwalletd path is unauthenticated and SPKI pinning is not actually enforced in the audited client path
5. Qortal redeem logic can overpay the miner by burning unintended change
6. release/update verification still trusts GitHub-hosted checksums without verifying artifact authenticity

## Findings

### High

#### 1. Mnemonic export protections are inconsistently enforced

The wallet exposes direct mnemonic export functions that bypass the documented warning, biometric, and passphrase-gated flow.

Affected locations:

- `crates/pirate-wallet-service/src/api.rs:1435`
- `crates/pirate-ffi-frb/src/api.rs:1156`
- `crates/pirate-cli-core/src/lib.rs:1002`

Impact:

- any unlocked caller with access to the service, FRB, or legacy CLI surface can extract the seed directly
- the legacy `seed` CLI command bypasses the safer multi-step export flow
- in multi-wallet sessions, the global seed-export state machine is not bound tightly enough to the wallet being exported

Risk:

- full wallet compromise

Team response:

- Semi false positive. Fixed. There was a genuine inconsistency which is fixed, but mnemonic protection is for the wallet only and it was incorrectly flagging the CLI & SDKs which are not supposed to have export protections

#### 2. Raw spending-key export is available without re-authentication

The service and FRB layers expose direct export of Sapling and Orchard spending keys without a second authentication step.

Affected locations:

- `crates/pirate-wallet-service/src/service.rs:109`
- `crates/pirate-wallet-service/src/service.rs:495`
- `crates/pirate-wallet-service/src/api/key_management.rs:78`
- `crates/pirate-ffi-frb/src/api.rs:1116`

Impact:

- any unlocked caller can export spend authority for the wallet or key group

Risk:

- full wallet compromise

Team response:

- Fixed

#### 3. Wallet secret storage can duplicate entries for the same wallet

`wallet_secrets.wallet_id` is the primary key, but the stored `wallet_id` value is encrypted with a fresh nonce on every write. That means the SQL upsert key changes every time, so `ON CONFLICT(wallet_id)` does not fire.

Affected locations:

- `crates/pirate-storage-sqlite/src/repository.rs:1257`
- `crates/pirate-storage-sqlite/src/repository.rs:1263`
- `crates/pirate-storage-sqlite/src/repository.rs:1292`
- `crates/pirate-storage-sqlite/src/repository.rs:1312`

Impact:

- repeated writes for the same logical wallet can leave multiple secret rows behind
- later reads scan and return the first decrypted match without deterministic ordering
- stale secret material can be selected after wallet updates, role changes, or migrations

Risk:

- nondeterministic key selection
- failed recovery
- watch-only/full-wallet state corruption

Team response:

- Fixed

#### 4. Deleted wallets can remain accessible through stale DB handles

Wallet deletion removes files and registry state, but thread-local cached database handles are not evicted before later reads.

Affected locations:

- `crates/pirate-wallet-service/src/api/encrypted_db.rs:417`
- `crates/pirate-wallet-service/src/api/wallet_registry.rs:357`

Impact:

- a previously opened wallet can remain readable on the same thread after deletion
- key export or other wallet access can continue even after the registry says the wallet is gone

Risk:

- post-deletion data exposure
- broken user expectations around secure wallet removal

Team response:

- Fixed

#### 5. Default lightwalletd traffic is unauthenticated and SPKI pinning is ineffective

The default client path uses plain HTTP to `64.23.167.130:9067`, and the audited client path does not perform an actual SPKI pin check before returning the connected channel.

Affected locations:

- `crates/pirate-sync-lightd/src/client.rs:37`
- `crates/pirate-sync-lightd/src/client.rs:42`
- `crates/pirate-sync-lightd/src/client.rs:1242`
- `app/lib/config/endpoints.dart:25`

Impact:

- sync and broadcast traffic can be intercepted or tampered with on the network path
- callers that believe they enabled SPKI pinning do not get the protection they expect

Risk:

- malicious chain data or response tampering
- privacy leakage
- transaction submission interference

Team response:

- Fixed

#### 6. Qortal redeem can burn excess value as miner fee

`build_qortal_p2sh_redeem_transaction` checks that the funding output covers outputs plus fee, but it never creates change. Any excess value becomes fee.

Affected locations:

- `crates/pirate-core/src/qortal_p2sh.rs:524`
- `crates/pirate-core/src/qortal_p2sh.rs:543`
- `crates/pirate-core/src/qortal_p2sh.rs:567`

Impact:

- if the funding output value is larger than the requested recipients plus intended fee, the remainder is silently overpaid to miners

Risk:

- direct fund loss

Team response:

- False positive

#### 7. Update and build verification trust GitHub release self-consistency, not authenticity

The updater and verify-build flow compare downloaded artifacts against published checksums from the same GitHub release. They do not establish artifact authenticity independently of the release host.

Affected locations:

- `app/lib/core/services/desktop_update_service.dart:256`
- `app/lib/features/settings/verify_build_screen.dart:193`
- `app/lib/features/settings/verify_build_screen.dart:601`

Impact:

- a compromised GitHub release, token, or maintainer can ship a malicious binary and matching checksum and still appear “verified”

Risk:

- supply-chain compromise

Team response:

- Fixed

#### 8. Unsigned Windows artifacts can be published and accepted by the updater

The release workflow explicitly falls back to publishing unsigned Windows artifacts if signing output is missing.

Affected locations:

- `.github/workflows/ci.yml:2202`
- `.github/workflows/ci.yml:2204`
- `.github/workflows/ci.yml:2210`
- `app/lib/core/services/desktop_update_service_asset_selection.dart:37`

Impact:

- official release assets can include unsigned Windows binaries
- the updater path can accept those unsigned assets

Risk:

- untrusted Windows update delivery

Team response:

- semi-false positive. It is intended for now to accept unsigned builds, however updater was patched so if you're on a signed version it won't accept to update to an unsigned version.

## Residual risk and coverage limits

- This was a source audit over the current working tree, not a clean tagged release.
- iOS and macOS runtime behavior were not exercised on local Apple hardware during this review.
- The review did not include live adversarial network testing or formal cryptographic verification.
- The codebase remains under active development, so findings should be tied to a frozen remediation snapshot before publication.
